# 類似ツールとの比較

このページでは ClaudeOS-Core と、同じ領域 (project-aware な Claude Code 設定) で動く他の Claude Code ツールを比較します。

**これは scope の比較で、品質の優劣ではありません。** 紹介するツールの多くは、それぞれの仕事で優れた働きをします。狙いは、ClaudeOS-Core が課題に合うのか、それとも別のツールのほうが合うのかを判断する助けにしてもらうことです。

> 英語原文: [docs/comparison.md](../comparison.md). 日本語訳は英語版に追従して同期されています。

---

## TL;DR

**実際のコードに基づいて `.claude/rules/` を自動生成したい** なら、ClaudeOS-Core の得意領域です。

それ以外 (広範なプリセットバンドル、計画ワークフロー、エージェントオーケストレーション、複数ツールの設定同期など) を求めているなら、Claude Code エコシステムの別のツールのほうが合います。

---

## ClaudeOS-Core が他ツールと違う点

ClaudeOS-Core を特徴づけるポイント:

- **実際のソースコードを読む** (deterministic な Node.js scanner で、LLM にスタックを推測させない)。
- **4-pass の Claude パイプライン** で事実注入型プロンプトを構成 (パスや慣習を一度抽出して再利用)。
- **5 つの post-generation validator**: `claude-md-validator` で構造、`content-validator` でパス記述とコンテンツ、`pass-json-validator` で中間 JSON、`plan-validator` で legacy plan ファイル、`sync-checker` で disk と sync-map の整合性をチェック。
- **10 言語の出力** に対応し、language-invariant な検証で品質を担保する。
- **プロジェクト固有の出力**: CLAUDE.md、`.claude/rules/`、standards、skills、guides、memory layer。すべてコードから導出し、プリセットバンドルから持ってこない。

同じ領域にある他の Claude Code ツール (重ねて使うのも、別のものを選ぶのもあり):

- **Claude `/init`**: Claude Code に組み込みで、1 回の LLM 呼び出しで単一の `CLAUDE.md` を生成する。小規模プロジェクトの素早い 1 ファイルセットアップに最適。
- **プリセット/バンドル系ツール**: 「ほとんどのプロジェクト」に通用するキュレート済みの agents、skills、rules を配布する。プロジェクトの慣習がバンドルのデフォルトと合うときに有効。
- **計画/ワークフロー系ツール**: フィーチャ開発向けに構造化された方法論 (specs、phases など) を提供する。Claude Code の上にプロセス層を載せたいときに合う。
- **Hook/DX 系ツール**: auto-save、コード品質フック、開発者体験の改善を Claude Code セッションに追加する。
- **クロスエージェントルール変換器**: Claude Code、Cursor などをまたいでルールを同期する。

これらのツールはたいてい **競合ではなく相補的** です。ClaudeOS-Core は「コードからプロジェクト固有ルールを生成する」仕事を担い、他のツールはまた別の仕事を担います。併用できるものも多いです。

---

## ClaudeOS-Core が合うとき

✅ Claude Code に一般的なやり方ではなく、プロジェクトの慣習に従わせたい。
✅ 新規プロジェクトを始める (もしくはチームをオンボードする) タイミングで、素早くセットアップしたい。
✅ コードベースの進化にあわせて `.claude/rules/` を手作業で保守するのに疲れた。
✅ [サポート済みの 12 スタック](stacks.md) のいずれかで作業している。
✅ deterministic で再現性のある出力が欲しい (同じコードなら毎回同じルール)。
✅ 英語以外の言語で出力したい (10 言語をビルトインでサポート)。

## ClaudeOS-Core が合わないとき

❌ スキャンステップなしで初日から動く、キュレート済みの agents/skills/rules バンドルが欲しい。
❌ スタックが未サポートで、コントリビュートする予定もない。
❌ エージェントオーケストレーション、計画ワークフロー、コーディング方法論が欲しい (それぞれに特化したツールを使うべき)。
❌ standards/rules/skills の完全セットではなく `CLAUDE.md` だけ欲しい (`claude /init` で十分)。

---

## scope での狭さ vs 広さ

ClaudeOS-Core は広範なバンドルより **狭い** (プリセットの agents、hooks、方法論は出荷せず、プロジェクト固有のルールだけを扱う)。一方で単一アーティファクトに絞ったツールより **広い** (CLAUDE.md だけでなく、standards、skills、guides、memory のマルチディレクトリツリーを生成)。プロジェクトでどちらの軸が重要かで選んでください。

---

## 「Claude /init で十分じゃない?」

もっともな疑問です。`claude /init` は Claude Code 組み込みで、1 回の LLM 呼び出しで単一の `CLAUDE.md` を生成します。速く、設定もいりません。

**うまくいくケース:**

- プロジェクトが小さい (30 ファイル以下)。
- ファイルツリーをざっと見て Claude にスタックを推測させることに抵抗がない。
- `.claude/rules/` のフルセットではなく、`CLAUDE.md` だけあればよい。

**苦戦するケース:**

- Claude が一見して認識できないカスタム慣習がある (例: JPA でなく MyBatis、独自のレスポンスラッパー、特殊なパッケージ構成)。
- チームメンバー間で再現性のある出力が欲しい。
- プロジェクトが大きすぎて、1 回の Claude 呼び出しでは解析を終える前にコンテキストウィンドウに当たる。

ClaudeOS-Core は `/init` が苦しい場面のために作られています。`/init` で目的が果たせるなら、ClaudeOS-Core は不要でしょう。

---

## 「ルールを手で書けばよくない?」

これももっともです。`.claude/rules/` の手書きは最も正確な選択肢で、自分のプロジェクトを一番よく知っているのは自分です。

**うまくいくケース:**

- プロジェクトが 1 つだけで、ソロ開発、ゼロからルールを書く時間が取れる。
- 慣習が安定していて、ドキュメント化も十分。

**苦戦するケース:**

- 新規プロジェクトを頻繁に立ち上げる (毎回ルール作成の時間が必要)。
- チームが大きくなり、ルールの中身を皆が忘れていく。
- 慣習が進化していて、ルールが追従できない。

ClaudeOS-Core は実用的なルールセットの大部分を 1 回の実行で揃えてくれます。残りは手動チューニング。多くのユーザーは、空のファイルから始めるよりこのほうが効率的だと感じています。

---

## 「プリセットバンドルとどう違う?」

Everything Claude Code のようなバンドルは、「ほとんどのプロジェクト」で動くキュレート済みのルール / スキル / エージェントセットを提供します。プロジェクトがバンドルの前提に合うなら、すばやく導入できます。

**バンドルがうまくいくケース:**

- プロジェクトの慣習がバンドルのデフォルトに合う (例: 標準的な Spring Boot や Next.js)。
- 通常から外れたツール選択がない (例: JPA でなく MyBatis)。
- 出発点が欲しいだけで、そこからカスタマイズする前提。

**バンドルが苦戦するケース:**

- スタックがデフォルト外のツールを使っている (バンドルの「Spring Boot」ルールは JPA 前提)。
- バンドルが知らない、強いプロジェクト固有の慣習がある。
- コードの進化にあわせてルールを更新したい。

ClaudeOS-Core はバンドルを補完できます: プロジェクト固有のルールに ClaudeOS-Core を使い、共通のワークフロールールにバンドルを重ねるとよいでしょう。

## 似たツール間で選ぶ

ClaudeOS-Core と別の project-aware Claude Code ツールで迷ったら、次を自問してください:

1. **コードを読ませたいか、それとも自分でプロジェクトを記述したいか?**
   コードを読ませる → ClaudeOS-Core。記述する → 多くの他ツール。

2. **毎回同じ出力が必要か?**
   はい → ClaudeOS-Core (deterministic)。いいえ → どれでも。

3. **standards/rules/skills/guides の完全セットが欲しいか、それとも CLAUDE.md だけか?**
   完全セット → ClaudeOS-Core。CLAUDE.md だけ → Claude `/init`。

4. **出力言語は英語か、他言語か?**
   英語だけ → 多くのツールが合う。他言語 → ClaudeOS-Core (10 言語ビルトイン)。

5. **エージェントオーケストレーション、計画ワークフロー、フックが必要か?**
   必要 → 専用ツールを使う。ClaudeOS-Core はこれらを扱いません。

ClaudeOS-Core を他のツールと並行して使った経験があれば、ぜひ [issue で報告](https://github.com/claudeos-core/claudeos-core/issues) してください。この比較をより正確にする助けになります。
