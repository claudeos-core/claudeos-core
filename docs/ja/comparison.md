# 類似ツールとの比較

このページは ClaudeOS-Core を、同じ一般領域 (project-aware な Claude Code 設定) で動作する他の Claude Code ツールと比較します。

**これは scope の比較であり、品質の判定ではありません。** 以下のツールの多くはそれぞれの仕事に優れています。要点は、ClaudeOS-Core があなたの問題に合うか、もしくは別のツールがより合うかを理解する手助けです。

> 英語原文: [docs/comparison.md](../comparison.md). 日本語訳は英語版に追従して同期されています。

---

## TL;DR

**コードに実際にあるものに基づいて `.claude/rules/` を自動生成したい** なら、それは ClaudeOS-Core の専門領域です。

それ以外のもの (広範なプリセットバンドル、計画ワークフロー、エージェントオーケストレーション、複数ツールの設定同期) を求めているなら、Claude Code エコシステムの別のツールがおそらくよりフィットします。

---

## ClaudeOS-Core が他ツールと違う点

ClaudeOS-Core を定義づける特徴:

- **実際のソースコードを読む** (deterministic な Node.js scanner — LLM がスタックを推測しない)。
- **4-pass の Claude パイプライン** で事実注入プロンプト (パス・コンベンションを一度抽出して再利用)。
- **5 つの post-generation validator** (`claude-md-validator` で構造、`content-validator` でパスクレームとコンテンツ、`pass-json-validator` で中間 JSON、`plan-validator` で legacy plan ファイル、`sync-checker` で disk ↔ sync-map の整合性)。
- **10 出力言語** で language-invariant な検証。
- **プロジェクト別出力**: CLAUDE.md、`.claude/rules/`、standards、skills、guides、memory layer — すべてあなたのコードから派生し、プリセットバンドルからではない。

この一般領域の他の Claude Code ツール (必要に応じて重ねたり、別のものを選んだり):

- **Claude `/init`** — Claude Code に組み込まれており、1 回の LLM 呼び出しで単一の `CLAUDE.md` を書く。小規模プロジェクトでの素早い 1 ファイルセットアップに最適。
- **プリセット/バンドル系ツール** — 「ほとんどのプロジェクト」に通用するキュレートされた agents、skills、rules を配布。あなたのコンベンションがバンドルのデフォルトに合うときに最適。
- **計画/ワークフロー系ツール** — フィーチャ開発のための構造化された方法論 (specs、phases など) を提供。Claude Code の上にプロセス層を載せたいときに最適。
- **Hook/DX 系ツール** — auto-save、コード品質フック、開発者体験の改善を Claude Code セッションに追加。
- **クロスエージェントルール変換器** — Claude Code、Cursor などをまたいでルールを同期。

これらのツールはほとんど **相補的で、競合的ではありません**。ClaudeOS-Core は「コードからプロジェクト別ルールを生成する」仕事を担当し、他は別の仕事を担当します。多くは併用可能です。

---

## ClaudeOS-Core が合うとき

✅ Claude Code に一般的なものではなく、あなたのプロジェクトのコンベンションに従わせたい。
✅ 新規プロジェクトを始める (またはチームをオンボードする) ところで、素早いセットアップが欲しい。
✅ コードベースの進化に合わせて `.claude/rules/` を手動メンテするのに疲れた。
✅ [サポートされている 12 スタック](stacks.md) のいずれかで作業している。
✅ deterministic で再現可能な出力が欲しい (同じコード → 毎回同じルール)。
✅ 英語以外の言語で出力が必要 (10 言語ビルトイン)。

## ClaudeOS-Core が合わないとき

❌ スキャンステップなしで初日から動く agents/skills/rules のキュレート済みプリセットバンドルが欲しい。
❌ スタックが未サポートで、コントリビュートする気がない。
❌ エージェントオーケストレーション、計画ワークフロー、コーディング方法論が欲しい — それらに特化したツールを使うべき。
❌ 標準/ルール/スキルの完全セットではなく `CLAUDE.md` だけが必要 — `claude /init` で十分。

---

## scope での狭さ vs 広さ

ClaudeOS-Core は広範なバンドルより **狭い** (プリセットの agents、hooks、方法論を出荷せず、あなたのプロジェクト固有のルールのみ)。単一アーティファクトに集中するツールよりは **広い** (CLAUDE.md に加え、standards、skills、guides、memory のマルチディレクトリツリーを生成)。プロジェクトでどの軸が重要かに基づいて選んでください。

---

## 「Claude /init で十分じゃない?」

正当な質問です。`claude /init` は Claude Code に組み込まれており、1 回の LLM 呼び出しで単一の `CLAUDE.md` を書きます。高速で設定不要です。

**うまくいくケース:**

- プロジェクトが小さい (≤30 ファイル)。
- ファイルツリーをざっと見て Claude にスタックを推測してもらうことに抵抗がない。
- フル `.claude/rules/` セットではなく、`CLAUDE.md` だけでよい。

**苦戦するケース:**

- プロジェクトが Claude が一見して認識しないカスタムコンベンションを持つ (例: JPA でなく MyBatis、カスタムレスポンスラッパー、特殊なパッケージレイアウト)。
- チームメンバー間で再現可能な出力が欲しい。
- プロジェクトが大きすぎて 1 回の Claude 呼び出しが解析を終える前にコンテキストウィンドウに当たる。

ClaudeOS-Core は `/init` が苦戦するケースのために作られています。`/init` であなたの目的が果たせるなら、ClaudeOS-Core はおそらく不要です。

---

## 「ルールを手で書けばよくない?」

これも正当です。`.claude/rules/` の手書きは最も正確な選択肢 — あなたが自分のプロジェクトを最もよく知っています。

**うまくいくケース:**

- プロジェクトが 1 つ、ソロ開発、ゼロからルールを書く時間をかけられる。
- コンベンションが安定していて十分にドキュメント化されている。

**苦戦するケース:**

- 新規プロジェクトを頻繁に始める (各プロジェクトでルール作成の時間がかかる)。
- チームが大きくなり、ルールの内容を皆が忘れる。
- コンベンションが進化し、ルールが追従できない。

ClaudeOS-Core は使えるルールセットの大部分を 1 回の実行で得させてくれます。残りは手動チューニング — そして多くのユーザーは、空のファイルから始めるよりこのほうが時間の使い方として良いと感じています。

---

## 「プリセットバンドルとどう違う?」

Everything Claude Code のようなバンドルは「ほとんどのプロジェクト」で動くキュレートされたルール / スキル / エージェントセットを与えてくれます。プロジェクトがバンドルの仮定に合うなら、迅速な採用に最適です。

**バンドルがうまくいくケース:**

- プロジェクトのコンベンションがバンドルのデフォルトに合う (例: 標準 Spring Boot や標準 Next.js)。
- 通常でないツール選択がない (例: JPA でなく MyBatis)。
- 出発点が欲しく、そこからカスタマイズしてよい。

**バンドルが苦戦するケース:**

- スタックがデフォルトでないツーリングを使う (バンドルの「Spring Boot」ルールは JPA を仮定)。
- バンドルが知らない強力なプロジェクト固有コンベンションがある。
- コードの進化に合わせてルールを更新したい。

ClaudeOS-Core はバンドルを補完できます: プロジェクト固有のルールに ClaudeOS-Core を使い、一般的なワークフロールールにバンドルを重ねる。

## 似たツール間で選ぶ

ClaudeOS-Core と別の project-aware Claude Code ツールの間で選ぶなら、以下を自問してください:

1. **コードを読ませたいか、プロジェクトを記述したいか?**
   コードを読む → ClaudeOS-Core。記述する → 多くの他ツール。

2. **毎回同じ出力が必要か?**
   はい → ClaudeOS-Core (deterministic)。いいえ → どれでも。

3. **完全な standards/rules/skills/guides が欲しいか、CLAUDE.md だけか?**
   完全セット → ClaudeOS-Core。CLAUDE.md だけ → Claude `/init`。

4. **出力言語は英語か、他言語か?**
   英語のみ → 多くのツールが合う。他言語 → ClaudeOS-Core (10 言語ビルトイン)。

5. **エージェントオーケストレーション、計画ワークフロー、フックが必要か?**
   はい → 適切な専用ツールを使う。ClaudeOS-Core はそれらを行いません。

ClaudeOS-Core と他のツールを並行して使ったことがあれば、体験を [issue で報告](https://github.com/claudeos-core/claudeos-core/issues) してください — この比較をより正確にするのに役立ちます。
