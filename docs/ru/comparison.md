# Сравнение со схожими инструментами

Здесь сравниваем ClaudeOS-Core с другими инструментами Claude Code из той же области (project-aware конфигурация Claude Code).

**Это сравнение по scope, а не оценка качества.** Большинство инструментов ниже отлично делают то, для чего созданы. Цель — понять, подходит ли ClaudeOS-Core под вашу задачу, или какой-то другой инструмент справится лучше.

> Английский оригинал: [docs/comparison.md](../comparison.md). Русский перевод синхронизирован с английским.

---

## TL;DR

Хотите **автоматически генерировать `.claude/rules/` на основе того, что реально лежит в коде**? Это специализация ClaudeOS-Core.

Нужно что-то другое (широкие preset-bundle, planning-workflow, оркестрация агентов, синхронизация конфигов между инструментами)? Тогда лучше посмотреть на другие инструменты экосистемы Claude Code.

---

## Чем ClaudeOS-Core отличается от других

Определяющие черты ClaudeOS-Core:

- **Читает реальный исходный код** (детерминированный Node.js-сканер, без LLM-угадывания стека).
- **4-pass конвейер Claude** с фактами, инжектируемыми в промпты (пути и конвенции извлекаются один раз и переиспользуются).
- **5 post-generation валидаторов**: `claude-md-validator` для структуры, `content-validator` для path-claim и контента, `pass-json-validator` для промежуточного JSON, `plan-validator` для legacy plan-файлов, `sync-checker` для консистентности disk ↔ sync-map.
- **10 выходных языков** с language-invariant валидацией.
- **Per-project вывод**: CLAUDE.md, `.claude/rules/`, standards, skills, guides, memory layer. Всё выведено из вашего кода, а не из preset-bundle.

Другие инструменты Claude Code в этой же области (можно комбинировать или выбирать по задаче):

- **Claude `/init`** — встроен в Claude Code, пишет один `CLAUDE.md` за один LLM-вызов. Подходит для быстрой одно-файловой настройки маленьких проектов.
- **Preset/bundle-инструменты** — распространяют курированные agents, skills или rules, рассчитанные на «большинство проектов». Подходят, когда ваши конвенции совпадают с дефолтами bundle.
- **Planning/workflow-инструменты** — дают структурированные методологии разработки фич (specs, фазы и т. д.). Подходят, когда нужен слой процесса поверх Claude Code.
- **Hook/DX-инструменты** — добавляют auto-save, code-quality хуки или DX-улучшения для сессий Claude Code.
- **Cross-agent rule converter** — синхронизируют rules между Claude Code, Cursor и т. д.

Эти инструменты в основном **дополняют друг друга, а не конкурируют**. ClaudeOS-Core отвечает за «генерацию per-project rules из вашего кода», остальные — за другие задачи. Большинство спокойно используется вместе.

---

## Когда ClaudeOS-Core подходит

✅ Хочется, чтобы Claude Code следовал конвенциям ВАШЕГО проекта, а не общим.
✅ Стартуете новый проект (или онбордите команду) и нужна быстрая настройка.
✅ Надоело вручную поддерживать `.claude/rules/` по мере эволюции кодовой базы.
✅ Стек входит в [12 поддерживаемых](stacks.md).
✅ Нужен детерминированный, повторяемый вывод (один и тот же код → одни и те же rules каждый раз).
✅ Нужен вывод не на английском (встроено 10 языков).

## Когда ClaudeOS-Core НЕ подходит

❌ Нужен курированный preset-bundle agents/skills/rules, работающий с первого дня без сканирования.
❌ Стек не поддерживается, и добавлять новый не хочется.
❌ Нужна оркестрация агентов, planning-workflow или coding methodology. Берите инструмент, заточенный под это.
❌ Достаточно одного `CLAUDE.md` без полного набора standards/rules/skills — хватит `claude /init`.

---

## Что у́же, а что шире по scope

ClaudeOS-Core **у́же**, чем broad-coverage bundle: не поставляет preset agents, hooks или methodology, только rules вашего проекта. И при этом **шире**, чем инструменты под один артефакт: генерирует CLAUDE.md плюс мульти-каталожное дерево standards, skills, guides и memory. Выбирайте по оси, которая важна для проекта.

---

## «Почему не использовать просто Claude /init?»

Справедливый вопрос. `claude /init` встроен в Claude Code и пишет один `CLAUDE.md` за один LLM-вызов. Быстро и без конфигурации.

**Хорошо работает, когда:**

- Проект маленький (≤30 файлов).
- Согласны, что Claude угадывает стек по быстрому осмотру дерева.
- Хватит одного `CLAUDE.md` без полного набора `.claude/rules/`.

**Не справляется, когда:**

- В проекте кастомная конвенция, которую Claude не распознаёт с быстрого взгляда (например, MyBatis вместо JPA, кастомная обёртка response, необычный layout пакетов).
- Нужен вывод, воспроизводимый между членами команды.
- Проект достаточно большой, чтобы один Claude-вызов упёрся в context window до завершения анализа.

ClaudeOS-Core сделан для случаев, где `/init` не справляется. Если `/init` устраивает — ClaudeOS-Core, скорее всего, не нужен.

---

## «Почему не написать rules вручную?»

Тоже справедливо. Ручное написание `.claude/rules/` — самый точный вариант: проект вы знаете лучше всех.

**Хорошо работает, когда:**

- Проект один, разработчик один, готовы потратить много времени на rules с нуля.
- Конвенции стабильны и хорошо задокументированы.

**Не справляется, когда:**

- Часто стартуете новые проекты, и каждый требует времени на rules.
- Команда растёт, люди забывают, что в rules.
- Конвенции эволюционируют, и rules отстают от них.

ClaudeOS-Core за один запуск доводит до почти готового rule-набора. Остаётся ручная подгонка, и многим это кажется лучшим использованием времени, чем старт с пустого листа.

---

## «В чём разница с preset-bundle?»

Bundle вроде Everything Claude Code дают курированный набор rules / skills / agents, рассчитанный на «большинство проектов». Хороши для быстрого внедрения, когда проект совпадает с допущениями bundle.

**Bundle хорошо работает, когда:**

- Конвенции проекта совпадают с дефолтами bundle (например, стандартный Spring Boot или стандартный Next.js).
- Нет необычных выборов стека (например, MyBatis вместо JPA).
- Нужна стартовая точка, от которой комфортно кастомизировать.

**Bundle не справляется, когда:**

- Стек использует не-default tooling (rules «Spring Boot» bundle ожидают JPA).
- В проекте сильная специфическая конвенция, о которой bundle не знает.
- Нужно, чтобы rules обновлялись по мере эволюции кода.

ClaudeOS-Core можно использовать как дополнение к bundle: ClaudeOS-Core отвечает за project-specific rules, а bundle накладывается сверху для общих workflow-rules.

## Выбор между похожими инструментами

При выборе между ClaudeOS-Core и другим project-aware инструментом Claude Code задайте себе вопросы:

1. **Хочу, чтобы инструмент читал код, или готов сам описать проект?**
   Чтение кода → ClaudeOS-Core. Описание → большинство остальных.

2. **Нужен один и тот же вывод каждый раз?**
   Да → ClaudeOS-Core (детерминированно). Нет → любой из них.

3. **Полный набор standards/rules/skills/guides или только один CLAUDE.md?**
   Полный набор → ClaudeOS-Core. Только CLAUDE.md → Claude `/init`.

4. **Английский язык вывода или другой?**
   Только английский → подойдёт много инструментов. Другие языки → ClaudeOS-Core (встроено 10 языков).

5. **Нужна оркестрация агентов, planning-workflow или хуки?**
   Да → берите специализированный инструмент. ClaudeOS-Core этим не занимается.

Если запускали ClaudeOS-Core и другой инструмент бок о бок, [откройте issue](https://github.com/claudeos-core/claudeos-core/issues) с опытом, это помогает делать сравнение точнее.
