# Сравнение со схожими инструментами

Эта страница сравнивает ClaudeOS-Core с другими инструментами Claude Code, работающими в той же общей области (project-aware конфигурация Claude Code).

**Это сравнение scope, а не суждение о качестве.** Большинство инструментов ниже отлично справляются со своим. Цель — помочь вам понять, подходит ли ClaudeOS-Core под вашу задачу или один из них подходит лучше.

> Английский оригинал: [docs/comparison.md](../comparison.md). Русский перевод синхронизирован с английским.

---

## TL;DR

Если вы хотите **автоматически генерировать `.claude/rules/` на основе того, что реально в вашем коде** — это специализация ClaudeOS-Core.

Если хотите чего-то другого (широкие preset-bundle-ы, planning workflow-ы, agent-оркестрация, синхронизация конфигов между инструментами) — другие инструменты в экосистеме Claude Code, вероятно, подойдут лучше.

---

## Чем ClaudeOS-Core отличается от других инструментов

Определяющие черты ClaudeOS-Core:

- **Читает ваш реальный исходный код** (детерминированный Node.js scanner — без LLM, угадывающего стек).
- **4-pass конвейер Claude** с инжектируемыми фактами в промптах (пути/конвенции извлекаются один раз и переиспользуются).
- **5 post-generation validator** (`claude-md-validator` для структуры, `content-validator` для path-claim и контента, `pass-json-validator` для промежуточного JSON, `plan-validator` для legacy plan-файлов, `sync-checker` для disk ↔ sync-map консистентности).
- **10 выходных языков** с language-invariant валидацией.
- **Per-project вывод**: CLAUDE.md, `.claude/rules/`, standards, skills, guides, memory layer — всё выведено из вашего кода, а не из preset-bundle.

Другие инструменты Claude Code в этом общем пространстве (вы можете слоить их или выбирать в зависимости от потребности):

- **Claude `/init`** — встроен в Claude Code; пишет один `CLAUDE.md` за один LLM-вызов. Лучше всего для быстрой одно-файловой настройки маленьких проектов.
- **Preset/bundle инструменты** — распространяют курированные agents, skills или rules, работающие для «большинства проектов». Лучше, когда ваши конвенции совпадают с defaults bundle.
- **Planning/workflow инструменты** — предоставляют структурированные методологии разработки фич (specs, фазы и т.д.). Лучше, когда нужен слой процесса поверх Claude Code.
- **Hook/DX инструменты** — добавляют auto-save, code-quality хуки или DX-улучшения для сессий Claude Code.
- **Cross-agent rule converter** — держат ваши rules синхронизированными между Claude Code, Cursor и т.д.

Эти инструменты в основном **дополняют, а не конкурируют**. ClaudeOS-Core отвечает за «генерация per-project rules из вашего кода»; другие — за другие задачи. Большинство можно использовать вместе.

---

## Когда ClaudeOS-Core — правильный выбор

✅ Хотите, чтобы Claude Code следовал конвенциям ВАШЕГО проекта, а не общим.
✅ Начинаете новый проект (или онбордите команду) и хотите быструю настройку.
✅ Устали вручную поддерживать `.claude/rules/` по мере эволюции кодовой базы.
✅ Работаете в одном из [12 поддерживаемых стеков](stacks.md).
✅ Нужен детерминированный, повторяемый вывод (тот же код → те же rules каждый раз).
✅ Нужен вывод не на английском (10 языков встроено).

## Когда ClaudeOS-Core НЕ правильный выбор

❌ Хотите курированный preset-bundle agents/skills/rules, работающий с первого дня без шага сканирования.
❌ Ваш стек не поддерживается, и контрибьютить новый вы не хотите.
❌ Хотите agent-оркестрацию, planning workflow или coding methodology — используйте инструмент, специализирующийся на этом.
❌ Нужен только один `CLAUDE.md`, а не полный набор standards/rules/skills — `claude /init` достаточно.

---

## Что у́же vs шире по scope

ClaudeOS-Core **у́же**, чем broad-coverage bundle-ы (он не поставляет preset agents, hooks или methodology — только rules вашего проекта). Он **шире**, чем инструменты, фокусирующиеся на одном артефакте (он генерирует CLAUDE.md плюс мульти-каталожное дерево standards, skills, guides и memory). Выбирайте по той оси, которая важна для вашего проекта.

---

## «Почему не использовать просто Claude /init?»

Справедливый вопрос. `claude /init` встроен в Claude Code и пишет один `CLAUDE.md` за один LLM-вызов. Быстро и без конфигурации.

**Хорошо работает, когда:**

- Проект маленький (≤30 файлов).
- Вы согласны на то, что Claude угадывает стек по быстрому осмотру дерева.
- Нужен только один `CLAUDE.md`, а не полный набор `.claude/rules/`.

**Не справляется, когда:**

- В проекте кастомная конвенция, которую Claude не распознаёт по быстрому взгляду (например, MyBatis вместо JPA, кастомная обёртка response, необычный layout пакетов).
- Нужен вывод, воспроизводимый между членами команды.
- Проект достаточно большой, чтобы один Claude-вызов упёрся в context window до завершения анализа.

ClaudeOS-Core построен для случаев, где `/init` не справляется. Если `/init` вам подходит — ClaudeOS-Core, скорее всего, не нужен.

---

## «Почему не написать rules вручную?»

Тоже справедливо. Ручное написание `.claude/rules/` — самый точный вариант: вы знаете свой проект лучше всех.

**Хорошо работает, когда:**

- У вас один проект, вы единственный разработчик, согласны потратить значительное время на написание rules с нуля.
- Конвенции стабильны и хорошо задокументированы.

**Не справляется, когда:**

- Вы часто стартуете новые проекты (каждый требует времени на rules).
- Команда растёт, и люди забывают, что в rules.
- Конвенции эволюционируют, и rules отстают от них.

ClaudeOS-Core доводит вас почти до годного rule-набора за один запуск. Остаток — ручная подгонка, и многим пользователям это кажется лучшим использованием времени, чем старт с пустого листа.

---

## «В чём разница vs использования preset-bundle?»

Bundle-ы вроде Everything Claude Code дают вам курированный набор rules / skills / agents, работающий для «большинства проектов». Они отличны для быстрого внедрения, когда ваш проект соответствует допущениям bundle.

**Bundle-ы хорошо работают, когда:**

- Конвенции вашего проекта совпадают с defaults bundle (например, стандартный Spring Boot или стандартный Next.js).
- У вас нет необычных выборов стека (например, MyBatis вместо JPA).
- Вам нужна стартовая точка, и вы рады кастомизировать оттуда.

**Bundle-ы не справляются, когда:**

- Стек использует не-default tooling (rules «Spring Boot» bundle-а предполагают JPA).
- У вас сильная project-specific конвенция, о которой bundle не знает.
- Хотите, чтобы rules обновлялись по мере эволюции кода.

ClaudeOS-Core может дополнять bundle-ы: используйте ClaudeOS-Core для project-specific rules; накладывайте bundle для общих workflow-rules.

## Выбор между похожими инструментами

Если выбираете между ClaudeOS-Core и другим project-aware инструментом Claude Code, спросите себя:

1. **Хочу ли я, чтобы инструмент читал мой код, или хочу описывать проект?**
   Чтение кода → ClaudeOS-Core. Описание → большинство других.

2. **Нужен ли мне один и тот же вывод каждый раз?**
   Да → ClaudeOS-Core (детерминированно). Нет → любой из них.

3. **Хочу полный набор standards/rules/skills/guides или только один CLAUDE.md?**
   Полный набор → ClaudeOS-Core. Только CLAUDE.md → Claude `/init`.

4. **Английский ли язык вывода или другой?**
   Только английский → подходит много инструментов. Другие языки → ClaudeOS-Core (10 языков встроено).

5. **Нужна ли agent-оркестрация, planning workflow или хуки?**
   Да → используйте подходящий специализированный инструмент. ClaudeOS-Core этим не занимается.

Если использовали ClaudeOS-Core и другой инструмент бок о бок, [откройте issue](https://github.com/claudeos-core/claudeos-core/issues) с вашим опытом — это помогает делать сравнение точнее.
