# Безопасность: что сохраняется при re-init

Частое опасение: *«Я кастомизировал свои `.claude/rules/`. Если снова запустить `npx claudeos-core init`, правки пропадут?»*.

**Короткий ответ:** зависит от того, используете ли `--force`.

Здесь разбираем, что именно происходит при перезапуске: что трогается, что нет.

> Английский оригинал: [docs/safety.md](../safety.md). Русский перевод синхронизирован с английским.

---

## Два пути через re-init

При перезапуске `init` на проекте, где уже есть выход, происходит одно из двух.

### Путь 1 — Resume (по умолчанию, без `--force`)

`init` читает существующие pass-маркеры (`pass1-*.json`, `pass2-merged.json`, `pass3-complete.json`, `pass4-memory.json`) в `claudeos-core/generated/`.

Для каждого pass-а: если маркер на месте и структурно валиден, pass **пропускается**. Если валидны все четыре маркера, `init` завершается рано: делать нечего.

**Эффект для ваших правок:** всё, что было отредактировано вручную, остаётся нетронутым. Ни один pass не выполняется, ни один файл не пишется.

Этот путь рекомендован для большинства сценариев в духе «я просто перепроверяю».

### Путь 2 — Чистый старт (`--force`)

```bash
npx claudeos-core init --force
```

`--force` удаляет pass-маркеры и rules, после чего запускает полный 4-pass конвейер с нуля. **Ручные правки rules теряются.** Это намеренно: `--force` — «эвакуационный люк» на случай «хочу чистую регенерацию».

Что удаляет `--force`:
- Все `.json` и `.md` файлы под `claudeos-core/generated/` (четыре pass-маркера + вывод сканера).
- Оставшийся каталог `claudeos-core/generated/.staged-rules/`, если предыдущий запуск упал посреди перемещения.
- Всё под `.claude/rules/`.

Что `--force` **не** удаляет:
- Файлы `claudeos-core/memory/` (decision log и failure patterns сохраняются).
- `claudeos-core/standard/`, `claudeos-core/skills/`, `claudeos-core/guide/` и т. д. Их перезаписывает Pass 3, но заранее не удаляет; всё, что Pass 3 не регенерирует, остаётся на месте.
- Файлы вне `claudeos-core/` и `.claude/`.
- CLAUDE.md (Pass 3 перезаписывает его в рамках обычной генерации).

**Почему `.claude/rules/` стирается под `--force`, а другие каталоги нет:** у Pass 3 есть guard «zero-rules detection», который срабатывает, когда `.claude/rules/` пуст; по нему решается, пропускать ли стадию per-domain rules. Если останутся устаревшие rules от предыдущего запуска, guard даст false-negative, и новые rules не сгенерируются.

## Почему `.claude/rules/` вообще существует (механизм staging)

Самый частый вопрос, поэтому отдельная секция.

В Claude Code есть **политика чувствительных путей**, которая блокирует записи в `.claude/` из subprocess-а, даже если он запущен с `--dangerously-skip-permissions`. Это намеренная safety-граница самого Claude Code.

Pass 3 и Pass 4 ClaudeOS-Core — это subprocess-вызовы `claude -p`, поэтому писать напрямую в `.claude/rules/` они не могут. Обходной путь:

1. Pass-prompt велит Claude писать все rule-файлы в `claudeos-core/generated/.staged-rules/`.
2. После завершения pass-а **Node.js orchestrator** (на который политика разрешений Claude Code *не* распространяется) обходит staging-дерево и переносит каждый файл в `.claude/rules/`, сохраняя sub-пути.
3. При полном успехе staging-каталог удаляется.
4. При частичной неудаче (file lock или ошибка cross-volume rename) staging-каталог **сохраняется**, чтобы можно было посмотреть, что не доехало; следующий запуск `init` повторит попытку.

Mover находится в `lib/staged-rules.js`. Сначала использует `fs.renameSync`, откатываясь на `fs.copyFileSync + fs.unlinkSync` при ошибках Windows cross-volume / antivirus file-lock.

**Что видно в реальности:** в нормальном потоке `.staged-rules/` создаётся и опустошается за один запуск `init`, можно вообще его не заметить. Если запуск упал посреди стадии, при следующем `init` файлы найдутся там, и `--force` их уберёт.

## Что когда сохраняется

| Категория файлов | Без `--force` | С `--force` |
|---|---|---|
| Ручные правки `.claude/rules/` | ✅ Сохранены (pass-ы не перезапускаются) | ❌ Потеряны (каталог стёрт) |
| Ручные правки `claudeos-core/standard/` | ✅ Сохранены (pass-ы не перезапускаются) | ❌ Перезаписаны Pass 3, если он регенерирует те же файлы |
| Ручные правки `claudeos-core/skills/` | ✅ Сохранены | ❌ Перезаписаны Pass 3 |
| Ручные правки `claudeos-core/guide/` | ✅ Сохранены | ❌ Перезаписаны Pass 3 |
| Ручные правки `CLAUDE.md` | ✅ Сохранены | ❌ Перезаписаны Pass 3 |
| Файлы `claudeos-core/memory/` | ✅ Сохранены | ✅ Сохранены (`--force` не удаляет память) |
| Файлы вне `claudeos-core/` и `.claude/` | ✅ Никогда не трогаются | ✅ Никогда не трогаются |
| Pass-маркеры (`generated/*.json`) | ✅ Сохранены (используются для resume) | ❌ Удалены (форсирует полный re-run) |

**Честное резюме:** в ClaudeOS-Core нет слоя diff-and-merge. Нет промпта «просмотрите изменения перед применением». История сохранения бинарная: либо перезапускаем только то, чего не хватает (по умолчанию), либо стираем и регенерируем (`--force`).

Если внесены значительные ручные правки, а нужно интегрировать новый сгенерированный контент, рекомендуемый workflow такой:

1. Сначала закоммитить правки в git.
2. Запустить `npx claudeos-core init --force` на отдельной ветке.
3. Через `git diff` посмотреть, что изменилось.
4. Вручную смержить нужное с обеих сторон.

Workflow намеренно громоздкий. Инструмент сознательно не пытается авто-мержить: ошибка тут незаметно испортила бы правила.

## Обнаружение pre-v2.2.0 апгрейда

Если запустить `init` на проекте с CLAUDE.md от более старой версии (pre-v2.2.0, до того как 8-секционный scaffold стал обязательным), инструмент это поймёт по числу заголовков (`^## ` heading count ≠ 8 — language-independent эвристика) и выдаст предупреждение:

```
⚠️  v2.2.0 upgrade detected
─────────────────────────
Your existing CLAUDE.md was generated with an older claudeos-core version.
v2.2.0 introduces structural changes that the default 'resume' mode
CANNOT apply because existing files are preserved under Rule B (idempotency).

To fully adopt v2.2.0, choose one of:
  1. Rerun with --force:   npx claudeos-core init --force
     (overwrites generated files; your memory/ content is preserved)
  2. Choose 'fresh' below  (equivalent to --force)
```

Предупреждение чисто информативное. Инструмент продолжает работать как обычно: хотите оставить старый формат — игнорируйте. Запустите с `--force`, и структурный апгрейд применится, а `claude-md-validator` пройдёт.

**Memory-файлы при `--force`-апгрейдах сохраняются.** Перезаписываются только сгенерированные файлы.

## Pass 4 immutability (v2.3.0+)

Тонкость: **Pass 4 не трогает `CLAUDE.md`.** Pass 3 уже прописал в Section 8 все нужные ссылки на L4 memory-файлы. Если бы Pass 4 тоже писал в CLAUDE.md, он бы повторно объявил содержимое Section 8 и породил ошибки валидатора `[S1]`/`[M-*]`/`[F2-*]`.

Это закреплено тремя способами:
- Промпт Pass 4 прямо говорит «CLAUDE.md MUST NOT BE MODIFIED».
- Функция `appendClaudeMdL4Memory()` в `lib/memory-scaffold.js` — no-op в 3 строки (всегда возвращает true, ничего не пишет).
- Регрессионный тест `tests/pass4-claude-md-untouched.test.js` фиксирует этот контракт.

**Что нужно знать пользователю:** если перезапускаете pre-v2.3.0 проект, где старый Pass 4 дописал Section 9 в CLAUDE.md, увидите ошибки `claude-md-validator`. Запустите `npx claudeos-core init --force`, чтобы регенерировать чисто.

## Что делает команда `restore`

```bash
npx claudeos-core restore
```

`restore` запускает `plan-validator` в режиме `--execute`. Исторически команда копировала содержимое из файлов `claudeos-core/plan/*.md` в локации, которые они описывают.

**Статус v2.1.0:** генерация master plan удалена в v2.1.0. `claudeos-core/plan/` больше не создаётся через `init` автоматически. Без файлов `plan/` `restore` превращается в no-op: пишет информационное сообщение и завершается чисто.

Команда оставлена для тех, кто вручную поддерживает plan-файлы для ad-hoc backup/restore. Если нужен реальный бэкап, используйте git.

## Паттерны восстановления

### «Я удалил какие-то файлы вне workflow ClaudeOS»

```bash
npx claudeos-core init --force
```

Перезапускает Pass 3 / Pass 4 с нуля. Удалённые файлы регенерируются. Ручные правки других файлов теряются (это же `--force`); для безопасности комбинируйте с git.

### «Хочу удалить конкретное правило»

Просто удалите файл. Следующий `init` (без `--force`) его не пересоздаст: resume-маркер Pass 3 пропустит весь pass.

Если хотите, чтобы файл пересоздался при следующем `init --force`, делать ничего не нужно: регенерация автоматическая.

Если хотите удалить навсегда (никогда не регенерировался), придётся держать проект пришпиленным к текущему состоянию и больше не запускать `--force`. Встроенного механизма «не регенерируй этот файл» нет.

### «Хочу постоянно кастомизировать сгенерированный файл»

В инструменте нет HTML-style begin/end маркеров для кастомных регионов. Два варианта:

1. **Не запускать `--force` на этом проекте.** При default-resume правки сохраняются бессрочно.
2. **Форкнуть prompt-шаблон.** Модифицируете `pass-prompts/templates/<stack>/pass3.md` в собственной копии инструмента, ставите свой форк, и регенерируемый файл отражает ваши кастомизации.

Для простых project-specific override-ов первого варианта обычно достаточно.

## Что проверяют валидаторы (после re-init)

После завершения `init` (резюме или `--force`) валидаторы запускаются автоматически:

- `claude-md-validator` — запускается отдельно через `lint`.
- `health-checker` — прогоняет четыре content/path-валидатора.

Если что-то не так (отсутствующие файлы, сломанные cross-reference, выдуманные пути), увидите вывод валидаторов. Список проверок см. в [verification.md](verification.md).

Валидаторы ничего не правят, только сообщают. Читаете отчёт и решаете: перезапускать `init` или править руками.

## Доверие через тесты

Путь «сохранить пользовательские правки» (resume без `--force`) покрыт интеграционными тестами в `tests/init-command.test.js` и `tests/pass3-marker.test.js`.

CI запускается на Linux / macOS / Windows × Node 18 / 20.

Если найдёте случай, когда ClaudeOS-Core потерял правки способом, противоречащим этому документу, — это баг. [Сообщите о нём](https://github.com/claudeos-core/claudeos-core/issues) с шагами воспроизведения.
