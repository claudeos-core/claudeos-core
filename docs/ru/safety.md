# Безопасность: что сохраняется при re-init

Распространённое опасение: *«Я кастомизировал свои `.claude/rules/`. Если я снова запущу `npx claudeos-core init`, потеряю ли я правки?»*

**Краткий ответ:** Зависит от того, используете ли вы `--force`.

Эта страница объясняет, что именно происходит при перезапуске: что трогается, что нет.

> Английский оригинал: [docs/safety.md](../safety.md). Русский перевод синхронизирован с английским.

---

## Два пути через re-init

Когда вы перезапускаете `init` на проекте, в котором уже есть выход, происходит одно из двух:

### Путь 1 — Resume (по умолчанию, без `--force`)

`init` читает существующие pass-маркеры (`pass1-*.json`, `pass2-merged.json`, `pass3-complete.json`, `pass4-memory.json`) в `claudeos-core/generated/`.

Для каждого pass-а: если маркер существует и структурно валиден, pass **пропускается**. Если все четыре маркера валидны, `init` завершается рано — ему нечего делать.

**Эффект на ваши правки:** всё, что вы вручную отредактировали, остаётся нетронутым. Ни один pass не выполняется, ни один файл не записывается.

Это рекомендуемый путь для большинства workflow-ов «я просто перепроверяю».

### Путь 2 — Чистый старт (`--force`)

```bash
npx claudeos-core init --force
```

`--force` удаляет pass-маркеры и rules, затем запускает полный 4-pass конвейер с нуля. **Ручные правки rules теряются.** Это намеренно — `--force` это «эвакуационный люк» для «хочу чистую регенерацию».

Что удаляет `--force`:
- Все `.json` и `.md` файлы под `claudeos-core/generated/` (четыре pass-маркера + выход сканера)
- Оставшийся каталог `claudeos-core/generated/.staged-rules/`, если предыдущий запуск упал в середине перемещения
- Всё под `.claude/rules/`

Что `--force` **не** удаляет:
- Файлы `claudeos-core/memory/` (ваш decision log и failure patterns сохраняются)
- `claudeos-core/standard/`, `claudeos-core/skills/`, `claudeos-core/guide/` и т.д. (они перезаписываются Pass 3, но не удаляются заранее — то, что Pass 3 не регенерирует, остаётся)
- Файлы вне `claudeos-core/` и `.claude/`
- Ваш CLAUDE.md (Pass 3 перезаписывает его в рамках обычной генерации)

**Почему `.claude/rules/` стирается под `--force`, а другие каталоги нет:** у Pass 3 есть guard «zero-rules detection», который срабатывает, когда `.claude/rules/` пуст; он используется для решения, пропускать ли стадию per-domain rules. Если есть устаревшие rules от предыдущего запуска, guard даст false-negative, и новые rules не сгенерируются.

## Почему `.claude/rules/` вообще существует (механизм staging)

Это самый частый вопрос, поэтому ему отдельная секция.

В Claude Code есть **политика чувствительных путей**, которая блокирует записи subprocess-ом в `.claude/`, даже когда subprocess запущен с `--dangerously-skip-permissions`. Это намеренная safety-граница в самом Claude Code.

Pass 3 и Pass 4 ClaudeOS-Core — это subprocess-вызовы `claude -p`, поэтому они не могут писать напрямую в `.claude/rules/`. Обходной путь:

1. Pass-prompt инструктирует Claude писать все rule-файлы в `claudeos-core/generated/.staged-rules/`.
2. После завершения pass-а **Node.js orchestrator** (на которого политика разрешений Claude Code *не* распространяется) обходит staging-дерево и переносит каждый файл в `.claude/rules/`, сохраняя sub-пути.
3. При полном успехе staging-каталог удаляется.
4. При частичной неудаче (file lock или ошибка cross-volume rename) staging-каталог **сохраняется**, чтобы вы могли посмотреть, что не доехало; следующий запуск `init` повторит попытку.

Mover находится в `lib/staged-rules.js`. Сначала использует `fs.renameSync`, откатываясь на `fs.copyFileSync + fs.unlinkSync` при ошибках Windows cross-volume / antivirus file-lock.

**Что вы реально видите:** в нормальном потоке `.staged-rules/` создаётся и опустошается за один запуск `init` — вы можете никогда его не заметить. Если запуск упадёт в середине стадии, вы найдёте файлы там при следующем `init`, и `--force` их уберёт.

## Что когда сохраняется

| Категория файлов | Без `--force` | С `--force` |
|---|---|---|
| Ручные правки `.claude/rules/` | ✅ Сохранены (pass-ы не перезапускаются) | ❌ Потеряны (каталог стёрт) |
| Ручные правки `claudeos-core/standard/` | ✅ Сохранены (pass-ы не перезапускаются) | ❌ Перезаписаны Pass 3, если он регенерирует те же файлы |
| Ручные правки `claudeos-core/skills/` | ✅ Сохранены | ❌ Перезаписаны Pass 3 |
| Ручные правки `claudeos-core/guide/` | ✅ Сохранены | ❌ Перезаписаны Pass 3 |
| Ручные правки `CLAUDE.md` | ✅ Сохранены | ❌ Перезаписаны Pass 3 |
| Файлы `claudeos-core/memory/` | ✅ Сохранены | ✅ Сохранены (`--force` не удаляет память) |
| Файлы вне `claudeos-core/` и `.claude/` | ✅ Никогда не трогаются | ✅ Никогда не трогаются |
| Pass-маркеры (`generated/*.json`) | ✅ Сохранены (используются для resume) | ❌ Удалены (форсирует полный re-run) |

**Честное резюме:** у ClaudeOS-Core нет слоя diff-and-merge. Нет prompt-а «просмотрите изменения перед применением». История сохранения бинарна: либо перезапустить только то, чего не хватает (по умолчанию), либо стереть и регенерировать (`--force`).

Если вы внесли значительные ручные правки и нужно интегрировать новый сгенерированный контент, рекомендуемый workflow:

1. Закоммитьте правки в git сначала.
2. Запустите `npx claudeos-core init --force` на отдельной ветке.
3. Используйте `git diff`, чтобы увидеть, что изменилось.
4. Вручную смержите то, что хотите, с обеих сторон.

Это намеренно громоздкий workflow. Инструмент сознательно не пытается авто-мержить — ошибка в этом тихо испортила бы правила тонкими способами.

## Обнаружение pre-v2.2.0 апгрейда

Когда вы запускаете `init` на проекте с CLAUDE.md, сгенерированным более старой версией (pre-v2.2.0, до того как 8-секционный scaffold стал enforced), инструмент обнаруживает это по числу заголовков (`^## ` heading count ≠ 8 — language-independent эвристика) и эмитит предупреждение:

```
⚠️  v2.2.0 upgrade detected
─────────────────────────
Your existing CLAUDE.md was generated with an older claudeos-core version.
v2.2.0 introduces structural changes that the default 'resume' mode
CANNOT apply because existing files are preserved under Rule B (idempotency).

To fully adopt v2.2.0, choose one of:
  1. Rerun with --force:   npx claudeos-core init --force
     (overwrites generated files; your memory/ content is preserved)
  2. Choose 'fresh' below  (equivalent to --force)
```

Предупреждение информативное. Инструмент продолжает нормально — если хотите оставить старый формат, можете игнорировать. Но при `--force` структурный апгрейд применяется, и `claude-md-validator` проходит.

**Memory-файлы сохраняются при `--force`-апгрейдах.** Перезаписываются только сгенерированные файлы.

## Pass 4 immutability (v2.3.0+)

Конкретная тонкость: **Pass 4 не трогает `CLAUDE.md`.** Section 8 Pass 3 уже написала все требуемые ссылки на L4 memory-файлы. Если бы Pass 4 тоже писал в CLAUDE.md, он бы повторно объявил содержимое Section 8, создавая ошибки validator `[S1]`/`[M-*]`/`[F2-*]`.

Это enforced двумя способами:
- Промпт Pass 4 явно говорит «CLAUDE.md MUST NOT BE MODIFIED».
- Функция `appendClaudeMdL4Memory()` в `lib/memory-scaffold.js` — это no-op в 3 строки (возвращает true безусловно, ничего не записывает).
- Регрессионный тест `tests/pass4-claude-md-untouched.test.js` enforced этот контракт.

**Что вам стоит знать как пользователю:** если перезапускаете pre-v2.3.0 проект, где старый Pass 4 дописал Section 9 в CLAUDE.md, увидите ошибки `claude-md-validator`. Запустите `npx claudeos-core init --force`, чтобы регенерировать чисто.

## Что делает команда `restore`

```bash
npx claudeos-core restore
```

`restore` запускает `plan-validator` в режиме `--execute`. Исторически она копировала содержимое из файлов `claudeos-core/plan/*.md` в локации, которые они описывают.

**Статус v2.1.0:** Генерация master plan была удалена в v2.1.0. `claudeos-core/plan/` больше не создаётся автоматически через `init`. Без файлов `plan/` `restore` — это no-op: логирует информационное сообщение и завершается чисто.

Команда оставлена для пользователей, вручную поддерживающих plan-файлы для ad-hoc backup/restore. Если нужен реальный backup, используйте git.

## Паттерны восстановления

### «Я удалил какие-то файлы вне workflow ClaudeOS»

```bash
npx claudeos-core init --force
```

Перезапускает Pass 3 / Pass 4 с нуля. Удалённые файлы регенерируются. Ваши ручные правки других файлов теряются (потому что `--force`) — для безопасности комбинируйте с git.

### «Хочу удалить конкретное правило»

Просто удалите файл. Следующий `init` (без `--force`) не пересоздаст его, потому что resume-маркер Pass 3 пропустит весь pass.

Если хотите, чтобы оно пересоздалось при следующем `init --force`, ничего делать не нужно — регенерация автоматическая.

Если хотите, чтобы оно было удалено навсегда (никогда не регенерировалось), нужно держать проект пришпиленным к текущему состоянию и не запускать `--force` снова. Встроенного механизма «не регенерируй этот файл» нет.

### «Хочу постоянно кастомизировать сгенерированный файл»

У инструмента нет HTML-style begin/end маркеров для кастомных регионов. Два варианта:

1. **Не запускать `--force` на этом проекте** — ваши правки сохраняются неограниченно при default-resume.
2. **Форкнуть prompt-шаблон** — модифицировать `pass-prompts/templates/<stack>/pass3.md` в собственной копии инструмента, установить ваш форк, и регенерируемый файл будет отражать ваши кастомизации.

Для простых project-specific override-ов варианта 1 обычно достаточно.

## Что проверяют validator (после re-init)

После завершения `init` (резюме или `--force`), validator запускаются автоматически:

- `claude-md-validator` — запускается отдельно через `lint`
- `health-checker` — запускает четыре content/path validator

Если что-то не так (отсутствующие файлы, сломанные cross-reference, выдуманные пути), вы увидите вывод validator. Список проверок см. в [verification.md](verification.md).

Validator ничего не исправляют — они сообщают. Вы читаете отчёт, затем решаете, перезапускать ли `init` или править вручную.

## Доверие через тестирование

Путь «сохранить пользовательские правки» (resume без `--force`) покрыт интеграционными тестами в `tests/init-command.test.js` и `tests/pass3-marker.test.js`.

CI запускается на Linux / macOS / Windows × Node 18 / 20.

Если найдёте случай, когда ClaudeOS-Core потерял ваши правки способом, противоречащим этому документу, — это баг. [Сообщите о нём](https://github.com/claudeos-core/claudeos-core/issues) с шагами воспроизведения.
